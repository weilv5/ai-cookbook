## 什么是 SSL/TLS

**SSL (Secure Sockets Layer，安全套接层)** 是网景公司在 1994 年推出的加密协议。

**TLS (Transport Layer Security，传输层安全)** 是 SSL 的后续标准化版本，现在通常说的 SSL 实际上大多指 TLS。

版本历史：

| SSL | 发布年份 | 状态 |
|-----|----------|------|
| SSL 1.0 | 1994 | 未公开发布 |
| SSL 2.0 | 1995 | 已废弃（不安全） |
| SSL 3.0 | 1996 | 已废弃（不安全） |
| TLS 1.0 | 1999 | 已废弃 |
| TLS 1.1 | 2006 | 已废弃 |
| TLS 1.2 | 2008 | 目前仍广泛使用 |
| TLS 1.3 | 2018 | 最新版本，推荐使用 |

> 现在主流网站都已经升级到 TLS 1.2 和 TLS 1.3，但习惯上我们还是叫它"SSL 证书"、"HTTPS SSL"。

## 为什么需要 SSL

在没有 SSL 的 HTTP 协议中，数据是**明文传输**的：

1. **窃听风险** — 第三方可以直接看到传输内容（密码、银行卡号等）
2. **篡改风险** — 第三方可以修改传输内容（插入广告、注入恶意代码）
3. **冒充风险** — 第三方可以冒充网站和你通信（钓鱼网站）

SSL 通过**加密**和**认证**解决了这三个问题：

- **加密**：就算数据被截获，对方也看不懂
- **认证**：证明你访问的确实是真实的网站
- **完整性**：防止数据被篡改

## SSL 工作原理

SSL 的核心是**非对称加密 + 对称加密**的组合使用。

### 两种加密方式对比

| 特性 | 对称加密 | 非对称加密 |
|------|----------|------------|
| 加密密钥 = 解密密钥 | ✅ 是 | ❌ 否 |
| 速度 | 快 | 慢 |
| 密钥交换 | 需要安全通道交换密钥 | 可以公开交换公钥 |
| 典型算法 | AES, ChaCha20 | RSA, ECC |

### 为什么不只用一种？

- 如果只用非对称加密：太慢了，不适合传输大量数据
- 如果只用对称加密：一开始怎么安全地交换密钥？

所以 SSL 采用了混合方案：

1. **握手阶段**：用非对称加密安全地协商出一个共享密钥（对称加密的密钥）
2. **数据传输阶段**：用协商好的对称密钥加密传输数据，速度快

## 完整握手过程

以最常见的 RSA 密钥交换为例：

### 四次握手流程

```
Client                                               Server

ClientHello
  (Cipher Suites, Compression Methods, Random)   →
                                                  ServerHello
                                               ←  (Cipher Suite, Compression Method, Random,
                                                    Server Certificate)

  PreMaster Secret (encrypted with server pubkey) →

[Client calculates Master Secret]                [Server calculates Master Secret]

Finished (encrypted with Master Secret)         →
                                                  Finished (encrypted with Master Secret)
                                               ←

Application Data                                  Application Data
(encrypted with session key)                 ↔  (encrypted with session key)
```

### 一步步详解

#### 第一步：客户端发起请求（ClientHello）

客户端发送：

- **支持的 TLS 版本**（比如 1.0, 1.1, 1.2, 1.3）
- **支持的加密套件列表**（Cipher Suite）
- **一个随机数 C**（Client Random）
- **压缩算法**

#### 第二步：服务器回应（ServerHello）

服务器选择双方都支持的配置：

- **选定的 TLS 版本**
- **选定的加密套件**
- **一个随机数 S**（Server Random）
- **服务器证书**（包含公钥，由 CA 签发）

> 💡 这里最重要的就是服务器证书，里面包含了服务器的公钥，而且有权威机构证明这个公钥确实属于这个域名。

#### 第三步：客户端验证证书并生成预主密钥

客户端做这些事情：

1. 验证服务器证书是否可信（是否由信任的 CA 签发、域名是否匹配、是否过期等）
2. 如果验证不通过，浏览器会弹出警告
3. 验证通过后，客户端生成**预主密钥（PreMaster Secret）**
4. 用服务器证书里的公钥加密预主密钥，发给服务器

#### 第四步：双方计算出会话密钥

服务器用自己的私钥解密，得到预主密钥。

现在客户端和服务器都有三个东西：
- 客户端随机数 C
- 服务器随机数 S
- 预主密钥 P

双方用相同的算法一起计算出**主密钥（Master Secret）**，再从中派生出：
- **对称加密密钥**（用于加密应用数据）
- **MAC 密钥**（用于验证完整性）

因为只有客户端和服务器才有这三个秘密，所以双方算出来的密钥是一样的。

#### 第五步：Finished 消息

客户端和服务器分别发送加密后的 Finished 消息，验证握手是否成功。

- 如果对方能正确解密，说明密钥协商成功
- 握手完成，开始传输应用数据

## SSL 流程图

```mermaid
flowchart TD
    A[客户端] -->|1. ClientHello<br/>(支持的加密套件、客户端随机数 C)| B[服务器]
    B -->|2. ServerHello<br/>(选中加密套件、服务器随机数 S、服务器证书)| A
    A -->|验证证书有效性| A
    A -->|3. 生成预主密钥 P<br/>用服务器公钥加密 P| B
    B -->|4. 用私钥解密得到 P| B
    A -->|5. C + S + P → 计算主密钥| A
    B -->|5. C + S + P → 计算主密钥| B
    A -->|6. 发送 Finished (加密)| B
    B -->|7. 发送 Finished (加密)| A
    A -->|握手成功<br/>开始对称加密通信| B
```

### TLS 1.3 改进的握手流程

TLS 1.3 把四次握手减少到一次，速度更快：

```mermaid
flowchart TD
    A[客户端] -->|ClientHello + KeyShare<br/>一次发送所有必要信息| B[服务器]
    B -->|ServerHello + Finished<br/>握手完成| A
    A -->|应用数据| B
    note right of A<br/>只需要 1-RTT<br/>比 1.2 快很多<br/>握手时间从 2-RTT → 1-RTT
```

## 常见概念辨析

### HTTP vs HTTPS

| 特性 | HTTP | HTTPS |
|------|------|-------|
| 端口 | 80 | 443 |
| 加密 | 明文 | TLS 加密 |
| 安全性 | 不安全 | 安全 |
| 证书 | 不需要 | 需要 CA 证书 |

### SSL 证书 vs 会话密钥

- **SSL 证书**：证明网站身份，里面包含公钥，一般有效期 1 年左右
- **会话密钥**：每次握手协商出来的对称加密密钥，每个连接一个，用完就扔

### 对称加密 vs 非对称加密 vs 哈希

- **对称加密**：加密解密用同一个密钥 → 用来加密大量数据
- **非对称加密**：公钥加密，私钥解密 → 握手阶段用来加密预主密钥
- **哈希函数**：生成摘要 → 验证数据完整性，也用于密钥派生

### 公钥 vs 私钥

- **公钥**：公开的，人人都能拿到 → 用来加密
- **私钥**：服务器偷偷保存 → 用来解密
- 数学关系：用公钥加密的只能用对应私钥解密，反过来也成立（签名的时候用私钥签名，公钥验证）

## 证书体系

### 证书链

浏览器为什么能相信服务器证书是真的？因为有**信任链**：

```
根 CA 证书（内置在浏览器/操作系统里）
    ↓ 签名
中间 CA 证书
    ↓ 签名
你的网站证书
```

浏览器只信任内置的根证书，如果根证书能给中间证书作证，中间证书能给你的证书作证，那就可信。

### 证书类型

按验证级别分：

1. **DV 域名验证**（Domain Validation）：只验证你拥有这个域名，最便宜，几分钟就能签发
2. **OV 组织验证**（Organization Validation）：验证你的组织身份，会显示公司名称，更可信，贵一些
3. **EV 扩展验证**（Extended Validation）：严格验证，地址栏会显示公司名，最贵

按覆盖域名分：

1. **单域名证书**：只覆盖一个域名
2. **多域名证书**：覆盖多个指定域名
3. **通配符证书**：覆盖 `*.example.com` 下所有子域名

### 如何获取免费证书

- [Let's Encrypt](https://letsencrypt.org/)：免费，90 天有效期，自动续期
- [ZeroSSL](https://zerossl.com/)：免费额度，支持通配符
- 各大云厂商也提供免费证书

## 性能优化

SSL 握手会增加延迟，怎么优化？

### 1. 会话复用（Session Resumption）

会话建立之后，下次再连接不用重新完整握手：

- **Session ID**：服务器给客户端一个 ID，存在客户端，下次连接带上，服务器找到之前协商好的密钥
- **Session Ticket**：服务器把会话状态加密发给客户端存着，下次连接带过来，不用服务器存状态

### 2. False Start

客户端在发送 ChangeCipherSpec 之后立即开始发送应用数据，不用等服务器的 Finished，减少一个 RTT。

### 3. TLS 1.3

直接把握手从 2-RTT 降到 1-RTT，0-RTT 甚至更快，而且加密更安全。

### 4. OCSP Stapling

证书吊销检查本来需要客户端去问 CA，现在服务器把吊销信息预先拿到，握手的时候发给客户端，节省一次网络请求。

### 5. HTTP/2 或 HTTP/3

HTTPS 配合 HTTP/2 多路复用，性能更好。HTTP/3 基于 QUIC，内置 TLS 1.3，更快。

## 常见问题

### Q: HTTPS 会比 HTTP 慢很多吗？

**A**: 握手阶段会增加几个 RT T 延迟，但握手只做一次，复用会话之后几乎不影响。启用了 HTTP/2 之后，很多场景下 HTTPS 反而比 HTTP 快。

### Q: 自签名证书能用吗？

**A**: 自己测试用可以，但公网上用户浏览器会报警告，因为浏览器不信任。不要用在公开网站上。

### Q: SSL 能保证绝对安全吗？

**A**: 没有绝对安全。正确使用 TLS 1.2+ 可以防止被动窃听和篡改，但如果你服务器私钥泄露了，或者证书被钓鱼偷了，那还是不安全。安全是整体的，SSL 是其中重要一环。

### Q: 为什么有时候握手失败？

常见原因：

- 证书过期了
- 证书域名不匹配
- 证书链不完整（服务器没发中间证书）
- 客户端不支持服务器选的加密套件
- 防火墙拦截了 443 端口

### Q: ECDHE 是什么？为什么比 RSA 好？

ECDHE 是椭圆曲线 Diffie-Hellman 密钥交换，支持**完美前向保密（PFS）**：就算服务器私钥后来泄露了，之前录下来的会话还是解不开。RSA 密钥交换不支持 PFS，如果私钥泄露，之前所有会话都能解密。所以现在都推荐用 ECDHE。

## 总结

SSL/TLS 的核心思路可以概括为：

1. **信任根** → 操作系统/浏览器预置可信 CA 根证书
2. **证书认证** → 服务器出示证书，证明自己身份和公钥
3. **密钥交换** → 用非对称加密安全协商出对称密钥
4. **对称加密** → 用协商好的密钥加密传输实际数据

这种设计兼顾了安全性和性能，是今天互联网安全通信的基础。
